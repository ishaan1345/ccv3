  ★ Insight ─────────────────────────────────────
  Key architectural observations from the deep exploration:
  1. August is a SPA (Single Page Application) built with React/Next.js.
  Client-side routing uses click handlers on divs, not <a> tags — which is why
  both the QA agent and automated clicks struggle. Tab clicks require targeting
  elements with [role="tab"].
  2. The CRM section is actually one page with tabs, not three separate pages.
  Companies/Contacts/Deals share a tabbed container at /connections*. The
  sidebar links just pre-select the tab.
  3. The "Add" asymmetry is by design: You can only ADD Contacts in August.
  Companies and Deals are read-only from HubSpot. This means August is primarily
   a consumer of HubSpot data, not a creator — except for contacts.
  4. The deal stage ID bug suggests August fetches pipeline stages from
  HubSpot's API but some custom/newer pipeline stages don't have their labels
  resolved. The raw IDs like 2640180956 are HubSpot internal pipeline stage
  object IDs.
  5. The [NaN:NaN] bug is specifically in the Overview tab's Outline section —
  the same conversation's Timeline tab renders timestamps correctly as [00:03].
  This means the data exists; the rendering code in the Outline component has a
  different timestamp parser that's broken.

  What Needs Correction

  1. "clicks through every page" is an overstatement

  The agent doesn't exhaustively click every button on every page. It runs 6
  specific test scenarios, each covering a feature area with step-by-step
  instructions. If a button isn't in the test prompt, it doesn't get clicked.
  The CATCH_ALL suffix helps catch visual anomalies beyond the explicit steps,
  but it can't catch bugs in interactions it never performs.

  2. Missing a known accuracy limitation

  The description should mention that browser-use has ~83% accuracy on complex 
  UIs (from CLAUDE.md Known Failure Modes). This isn't a minor caveat — it means
   roughly 1 in 6 actions may fail or misfire. This is why max_actions: 10 is
  kept low in config.yaml — more actions = more chances for the agent to go
  off-rails.

  3. My report had an error about Search bars — which affects AUGUST_CONTEXT.md 
  accuracy

  In my comprehensive report, I said "No visible Search bar" on
  Companies/Contacts/Deals. That was wrong. Search bars DO exist on those pages
  — they're <input> textboxes, not <button> elements, so my button-scanning
  script missed them. From my own ARIA snapshot of the Deals page:
  textbox "Search" [ref=e109]:
    /placeholder: Search Deals
  And from the Conversations page:
  textbox [ref=e101]:
    /placeholder: Search Conversations

  So the original AUGUST_CONTEXT.md was correct about search existing, and the
  other agent's "correction" removing search from the spec was actually wrong.
  Search bars exist on all CRM pages.

  4. The "no selectors to maintain" claim is mostly true but nuanced

  The agent doesn't use CSS selectors, but it IS bounded by the quality of
  AUGUST_CONTEXT.md. If the spec describes a feature incorrectly (like claiming
  a Coaching tab exists when it doesn't), the agent will try to find that
  non-existent feature and report a false failure. So there ARE things to
  maintain — they're just natural language descriptions instead of CSS
  selectors. Different maintenance burden, not zero maintenance.

  ★ Insight ─────────────────────────────────────
  The fundamental tradeoff of vision-based QA vs. selector-based QA:
  - Selector tests: brittle to UI changes, but 100% reliable when they work. A
  cy.get('.btn-sync').click() either finds the button or it doesn't — no
  ambiguity.
  - Vision tests: resilient to UI changes, but ~83% reliable per action. The
  agent "sees" the Sync button regardless of its CSS class, but it might click
  the wrong spot, get confused by overlapping elements, or hallucinate a
  success.
  - The real risk: a vision-based agent can see a page and report PASSED even if
   something is subtly wrong (wrong data, missing field) if the spec doesn't
  specifically call it out. Selector tests at least throw explicit errors on
  structural mismatches.

  The description is a solid, accurate overview. Fix the 4 issues above and it's
